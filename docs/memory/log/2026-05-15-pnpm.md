# 2026-05-15 - Switch from npm to pnpm

## Change

- `package-lock.json` deleted; `pnpm-lock.yaml` is now the lockfile
- `bin/setup` runs `pnpm install` (was `npm install` in the old `make deps` target)
- `package.json` pins `"packageManager": "pnpm@11.0.3"` (matches the version that wrote the lockfile)
- CI workflows use `pnpm/action-setup@0e279bb` (v6.0.8) with no explicit `version:` — the action reads `packageManager` from `package.json`, so the field is the single source of truth for both local and CI
- `package.json`'s `checkly` + `jiti` + `typescript` devDeps stay pinned to exact versions

## Why

Pnpm is more secure, and faster.

## On corepack

Node 24 ships without corepack as a default-on package manager bridge, so we don't need a `corepack enable` step. The `packageManager` field is still load-bearing without corepack on two paths:

1. **Locally**: pnpm 11 honors `packageManager` via the `pmOnFail` setting (default `download`). On a version mismatch, pnpm downloads the pinned version and re-execs. ([pnpm settings docs](https://pnpm.io/settings#pmonfail) — this replaces the old `manage-package-manager-versions` setting that was removed in v11.)
2. **In CI**: `pnpm/action-setup` reads the field when no `version:` input is provided. Documented at [pnpm/action-setup#inputs](https://github.com/pnpm/action-setup#inputs): "Version… Optional when there is a `packageManager` field in the `package.json`."
